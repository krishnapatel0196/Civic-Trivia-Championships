Add two new endpoints to the existing admin router in backend/src/routes/admin.ts:

**1. GET /flags — Paginated list of flagged questions**

Query params:
- `page` (default 1), `limit` (default 25, max 100)
- `sort`: 'flag_count' | 'created_at' (default: 'flag_count')
- `order`: 'asc' | 'desc' (default: 'desc')
- `collection`: collection slug filter (optional)
- `tab`: 'active' | 'archived' (default: 'active')

Implementation:
- When tab=active: filter `questions.status = 'active' AND questions.flagCount > 0`
- When tab=archived: filter `questions.status = 'archived'`
- LEFT JOIN collectionQuestions + collections to get collection names (same pattern as /questions/explore)
- For each question, also query question_flags to aggregate reason counts. Use a subquery or a second query to get reason breakdowns. The reasons column is jsonb array of strings like ['confusing-wording', 'wrong-answer']. Aggregate across all flags for each question to produce a breakdown object like `{ 'confusing-wording': 3, 'wrong-answer': 2 }`.
- Also include `lastFlaggedAt` — the MAX(created_at) from question_flags for each question.
- Truncate question text to 120 chars for table display (same as /questions/explore).
- Return shape: `{ data: FlaggedQuestionRow[], pagination: { page, limit, total, totalPages } }`
- FlaggedQuestionRow: `{ id, externalId, text, difficulty, status, flagCount, collectionNames, reasonBreakdown, lastFlaggedAt }`
- Use denormalized flag_count for sorting — do NOT use COUNT(*) from question_flags for the sort. The reason breakdown is a secondary query/subquery for display only.
- Apply collection filter by joining on collections.slug if provided.
- IMPORTANT: Reset page to bounds if out of range.
- Use the same count query pattern as /questions/explore for pagination total.

**2. GET /flags/:questionId/detail — Full question context + individual flags**

Path param: questionId (numeric question ID)

Implementation:
- Fetch full question data: id, externalId, text, options, correctAnswer, explanation, difficulty, status, source, learningContent, flagCount, collectionNames (same shape as /questions/:id/detail).
- Fetch individual flags from question_flags WHERE questionId = :questionId, ordered by created_at DESC, limited to 20 most recent.
- For each flag, JOIN to auth.users table to get username/display_name. The users table is in the default schema (not civic_trivia), so use raw SQL for the join: `LEFT JOIN auth.users ON question_flags.user_id = auth.users.id`. Get the user's `name` or `email` field.
- Return shape: `{ question: QuestionDetail, flags: FlagEntry[] }`
- FlagEntry: `{ id, userId, username, reasons, elaborationText, createdAt }`
- Handle "Unknown User" gracefully if user record not found (LEFT JOIN).

Import `questionFlags` from schema.ts at the top of the file (it's already exported but not imported in admin.ts).

IMPORTANT: The users table is NOT in the civic_trivia schema — it's in the public/auth schema. Check how other routes reference users. If there's no Drizzle table definition for users, use raw SQL via `sql` template for the JOIN. Look at how middleware/auth.ts or other files reference the users table to determine the correct table name and schema.

Add three action endpoints to the admin router. Note: an archive endpoint already exists at `POST /questions/:id/archive` — but it does NOT clear flag_count. Create a new version or modify the existing one for the flag review use case.

**1. PATCH /flags/:questionId/archive — Archive a flagged question**

Implementation:
- Parse questionId as integer, validate.
- In a transaction:
  - Update questions SET status = 'archived', flagCount = 0, updatedAt = NOW()
  - Append to expirationHistory jsonb array: `{ action: 'archived', timestamp: ISO8601 }`
- Return `{ success: true }`
- Use the EXISTING archive pattern from `POST /questions/:id/archive` but add flagCount = 0 reset.

Actually — to avoid confusion with the existing `POST /questions/:id/archive` endpoint, put these under the `/flags/` prefix to keep flag review actions namespaced:
- `PATCH /flags/:questionId/archive`
- `POST /flags/:questionId/dismiss`
- `PATCH /flags/:questionId/restore`

**2. POST /flags/:questionId/dismiss — Dismiss flags (keep question active)**

Implementation:
- Parse questionId as integer, validate.
- In a transaction:
  - Delete all rows from question_flags WHERE questionId = :questionId
  - Update questions SET flagCount = 0, updatedAt = NOW() WHERE id = :questionId
- Return `{ success: true }`
- This is a hard delete of flag records (per RESEARCH recommendation — simpler, no audit trail needed).

**3. PATCH /flags/:questionId/restore — Restore an archived question**

Implementation:
- Parse questionId as integer, validate.
- Verify question exists and status = 'archived'.
- Update questions SET status = 'active', updatedAt = NOW() WHERE id = :questionId
- Append to expirationHistory: `{ action: 'renewed', timestamp: ISO8601 }` (use 'renewed' to match existing history format).
- Return `{ success: true }`

All three endpoints are protected by the router-level `authenticateToken, requireAdmin` middleware already applied at the top of admin.ts.

    **authStore.ts — extend interface and implementation:**

    Add three new fields and actions to the AuthStore interface:
    - `tierResolved: boolean` (default `false`) — tracks whether the authoritative tier has been fetched
    - `setTier: (tier: Tier) => void` — sets `tier` field (follows `setDisplayName` pattern)
    - `setTierResolved: (resolved: boolean) => void` — sets `tierResolved` field

    In the `create()` call:
    - Add `tierResolved: false` to initial state
    - Add `setTier: (tier: Tier) => set({ tier })` action
    - Add `setTierResolved: (resolved: boolean) => set({ tierResolved: resolved })` action

    In the `clearAuth` action, add `tierResolved: false` to the reset object (alongside existing `tier: null`).

    Do NOT add `isAdmin` field — research recommends deferring that. Do NOT change `setAuth` behavior. Do NOT modify the `user` object shape.

    **AuthInitializer.tsx — add post-restore tier resolution:**

    Import `fetchAccountProfile` from `'../services/accountsApi'` (already imported `exchangeRefreshToken` from there).

    In the `initializeAuth` function, after the successful session restore block (after `setAuth(data.access_token, {...})`), add a try/catch/finally block that:
    1. Calls `await fetchAccountProfile(data.access_token)` to get authoritative tier
    2. On success: calls `useAuthStore.getState().setTier(profile.tier)` and `useAuthStore.getState().setDisplayName(profile.display_name)`
    3. On catch: silently ignores — session is valid, tier stays at whatever token metadata said
    4. In finally: calls `useAuthStore.getState().setTierResolved(true)`

    IMPORTANT: Move `setLoading(false)` from the existing `finally` block into the NEW finally block (after fetchAccountProfile completes). This prevents the brief flash where AdminGuard sees stale tier. The existing `finally { setLoading(false) }` should be removed and replaced. The flow becomes:
    - `setAuth(...)` — sets isAuthenticated, user, initial tier
    - `fetchAccountProfile(...)` — resolves authoritative tier
    - `setTierResolved(true)` + `setLoading(false)` — in the new finally block

    In the `else` branch (exchangeRefreshToken returned null — no valid session):
    - After `clearAuth()`, add `useAuthStore.getState().setTierResolved(true)` — no session means tier is resolved as null

    In the `catch` branch (unexpected error):
    - After `clearAuth()`, add `useAuthStore.getState().setTierResolved(true)`

    In the `!storedRefresh` early return:
    - After `setLoading(false)`, add `useAuthStore.getState().setTierResolved(true)` — or set it before setLoading. Either order is fine since there's no session.

    Remove the outer `finally { setLoading(false) }` block entirely — `setLoading(false)` is now called in the specific branches above.

    Update the useEffect dependency array to include any new store getters if destructured (likely none needed — `setTierResolved` is accessed via `useAuthStore.getState()`, not destructured).

    **App.tsx — update AdminGuard to respect tierResolved:**

    In the `AdminGuard` function:
    - Destructure `tierResolved` and `tier` from `useAuthStore()` (add to existing destructuring)
    - Change the loading check from `if (isLoading) return null` to `if (isLoading || !tierResolved) return null`
    - Change the admin check from `if (!user || user.tier !== 'empowered')` to `if (!user || tier !== 'empowered')` — use the top-level store `tier` field (which is the authoritative value after resolution), NOT `user.tier` (which may be stale from the JWT metadata)

    Do NOT change AdminGuard's rendering structure beyond these checks. Keep `<Navigate to="/login">` and `<Forbidden />` as-is.

    This ensures that navigating to Profile always updates the store with the authoritative tier from the accounts API. Import `setTier` is not needed — it's accessed via `useAuthStore.getState()`.

    **Fix 2: Guard accessToken before calling fetchAccountProfile.**
    Replace the current `fetchAccountProfile(accessToken ?? '')` call with a proper null guard.

    At the top of the `load()` function, after `const { accessToken } = useAuthStore.getState();`, add:
    ```typescript
    if (!accessToken) {
      navigate('/login?from=/profile', { replace: true });
      return;
    }
    ```

    Then change the `Promise.allSettled` array to pass `accessToken` directly (no `?? ''` fallback):
    ```typescript
    const [triviaResult, accountResult] = await Promise.allSettled([
      fetchTriviaStats(),
      fetchAccountProfile(accessToken),
    ]);
    ```

    This prevents sending `Authorization: Bearer ` with an empty string. The ProtectedRoute should prevent this path, but the guard is defense-in-depth.

    Do NOT change any other Profile.tsx behavior — leave stats rendering, timer multiplier, TierBadge, and error handling as-is.

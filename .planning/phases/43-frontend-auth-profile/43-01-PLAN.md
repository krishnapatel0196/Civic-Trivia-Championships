<objective>
Rewire the frontend auth infrastructure to call the Empowered Accounts API instead of local backend auth routes. This replaces the entire auth plumbing: types, services, store, token refresh interceptor, and startup initializer.

Purpose: The trivia frontend currently authenticates against its own backend's /auth/* routes using cookie-based sessions. This plan switches all auth calls to the shared Empowered Accounts API, using hybrid token storage (access token in Zustand memory, refresh token in localStorage).

Output: Updated auth types, new accountsApi.ts service, rewritten authService.ts, updated authStore.ts, rewritten api.ts interceptor, rewritten AuthInitializer.tsx, updated .env files.
</objective>

**frontend/src/services/accountsApi.ts** — Create new file. This is a dedicated fetch wrapper for the Empowered Accounts API only. Do NOT use apiRequest from api.ts (that's for the trivia backend).
- Export `ACCOUNTS_API_URL = import.meta.env.VITE_EMPOWERED_ACCOUNTS_URL || 'http://localhost:3001'`.
- Export `accountsApiFetch<T>(path: string, options?: RequestInit): Promise<T>` — a simple fetch wrapper that:
  - Prepends `ACCOUNTS_API_URL` to path
  - Sets `Content-Type: application/json` header
  - Does NOT set `credentials: 'include'` (causes CORS errors against accounts API)
  - On non-OK response, parses JSON error body and throws it
  - On OK response, returns parsed JSON
- Export `exchangeRefreshToken(): Promise<{ access_token: string; refresh_token: string; expires_in: number; user: AccountsUser } | null>`:
  - Reads refresh token from `localStorage.getItem('ev_refresh_token')`
  - If no token, return null immediately
  - Calls Supabase native token refresh: `POST ${import.meta.env.VITE_SUPABASE_URL || ACCOUNTS_API_URL}/auth/v1/token?grant_type=refresh_token` with body `{ refresh_token }` and header `apikey: ${import.meta.env.VITE_SUPABASE_ANON_KEY}`. Note: Supabase native refresh returns `{ access_token, refresh_token, expires_in, user: { id, email, ... } }`.
  - IMPORTANT: The Supabase native refresh returns user in a different shape than the accounts API login. Map the response: extract `user.id`, `user.email`. For tier, it won't be in the Supabase response — set tier to null (it will be resolved later by profile fetch or can remain from prior login state).
  - On success, store new refresh_token in localStorage (`ev_refresh_token`), return the mapped response.
  - On failure (any error), remove `ev_refresh_token` from localStorage, return null.
  - ALTERNATIVE simpler approach if VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY are not available: call `POST ${ACCOUNTS_API_URL}/auth/v1/token?grant_type=refresh_token` with the refresh_token body. Try this path first. If the accounts API proxies Supabase auth, this will work.

**frontend/src/services/authService.ts** — Rewrite to use accountsApiFetch instead of apiRequest:
- Import `accountsApiFetch` from `./accountsApi`.
- Import `AuthResponse, LoginCredentials, SignupData, AccountsUser` from `../types/auth`.
- `signup(data: SignupData)`: POST `/api/auth/signup` via accountsApiFetch. Body: `{ email, password }` (no name field). Returns `Promise<{ message: string }>` (signup does not auto-login on accounts API).
- `login(credentials: LoginCredentials)`: POST `/api/auth/login` via accountsApiFetch. Returns `Promise<AuthResponse>` (with access_token, refresh_token, user).
- `logout(accessToken: string)`: POST `/api/auth/logout` via accountsApiFetch with `Authorization: Bearer ${accessToken}` header. Returns `Promise<{ message: string }>`. Always resolves (per integration guide, always 200).
- `fetchAccountProfile(accessToken: string)`: GET `/api/account/me` via accountsApiFetch with Bearer token. Returns the full account profile (display_name, tier, connected_profile with xp/gem_balance, avatar_url, etc.). Type the response as `AccountProfile` — define this interface inline or in types/auth.ts with fields: `{ id: string; email: string; display_name: string; avatar_url: string | null; tier: Tier; account_standing: string; connected_profile?: { xp: number; gem_balance: number; display_name: string; } }`.

**frontend/src/services/api.ts** — Replace cookie-based refresh with localStorage-based refresh:
- Import `exchangeRefreshToken` from `./accountsApi`.
- Import `useAuthStore` (keep existing).
- Remove the `refreshToken()` function entirely.
- In `apiRequest()`:
  - Remove `credentials: 'include'` from defaultOptions. The trivia backend now validates JWTs from the Authorization header, not cookies.
  - Keep `Content-Type: application/json`.
  - Add: Before making the request, read `accessToken` from `useAuthStore.getState()`. If accessToken exists, add `Authorization: Bearer ${accessToken}` to merged headers automatically. This removes the need for callers to manually add the auth header.
  - On 401 response (when not already refreshing):
    - Set `isRefreshing = true`
    - Call `exchangeRefreshToken()` from accountsApi
    - If returns data: call `useAuthStore.getState().setAuth(data.access_token, data.user)`, store `data.refresh_token` in localStorage `ev_refresh_token`, retry original request with new token
    - If returns null: call `useAuthStore.getState().clearAuth()`, throw session expired error
    - Ensure `finally { isRefreshing = false }` wraps the refresh block

**frontend/src/components/AuthInitializer.tsx** — Rewrite startup check:
- Import `exchangeRefreshToken` from `../services/accountsApi`.
- Import `useAuthStore` (keep existing).
- Remove import of `authService`.
- In `useEffect`:
  1. Check `localStorage.getItem('ev_refresh_token')` — if null/empty, call `clearAuth()` immediately and `setLoading(false)`. No network call. Return early.
  2. If refresh token exists, call `exchangeRefreshToken()`.
  3. If returns data: store `data.refresh_token` in localStorage `ev_refresh_token`, call `setAuth(data.access_token, { id: data.user?.id || '', email: data.user?.email || '', tier: data.user?.tier || 'inform' })`. Note: tier may not be in the Supabase refresh response — pass what's available, the profile fetch on the Profile page will get the full data.
  4. If returns null: call `clearAuth()` (refresh token was invalid/expired).
  5. Always `setLoading(false)` in finally block.
- Keep the existing loading spinner UI unchanged.

  <name>Task 1: Create and run the content migration script</name>
  <files>backend/src/scripts/migrate-to-shared-supabase.ts</files>
  <action>
    Create `backend/src/scripts/migrate-to-shared-supabase.ts` that:

    1. Uses two `pg.Pool` connections:
       - `SOURCE_DATABASE_URL` (current dev Supabase — the `civic_trivia` schema)
       - `TARGET_DATABASE_URL` (shared Supabase — the `trivia` schema)
       Both env vars must be set. If either is missing, exit with a clear error message.

    2. Migrates tables IN THIS ORDER (respects FK dependencies):
       a. `topics` (no FKs)
       b. `collections` (no FKs)
       c. `election_races` (no FKs)
       d. `questions` (FK to topics, FK to election_races)
       e. `collection_topics` (FK to collections, FK to topics)
       f. `collection_questions` (FK to collections, FK to questions)

    3. Does NOT migrate:
       - `question_flags` (old integer user_id incompatible with UUID; start fresh)
       - `player_stats` (new table, starts empty)
       - `player_prefs` (new table, starts empty)

    4. For each table:
       a. `SELECT * FROM civic_trivia.{table} ORDER BY id` (or by composite PK for junction tables)
       b. Batch insert into `trivia.{table}` using parameterized queries, batch size 500
       c. Use `INSERT ... ON CONFLICT DO NOTHING` for idempotency
       d. Log: table name, source row count, inserted count

    5. After all tables migrated, run verification queries:
       a. For each table, compare source count vs target count
       b. Log a summary table showing counts
       c. Exit with code 0 if all counts match, code 1 if any mismatch

    6. Handle SERIAL sequences: After inserting into tables with SERIAL PKs (topics, collections, election_races, questions), reset the sequence to max(id) + 1:
       ```sql
       SELECT setval(pg_get_serial_sequence('trivia.{table}', 'id'), (SELECT COALESCE(MAX(id), 0) FROM trivia.{table}));
       ```
       This ensures future inserts get correct auto-increment values.

    7. Clean up: Close both pools at end.

    Run the script: `cd backend && npx tsx src/scripts/migrate-to-shared-supabase.ts`

    If SOURCE_DATABASE_URL or TARGET_DATABASE_URL are not set, ask the user to provide them. SOURCE_DATABASE_URL is the current dev project's direct Postgres connection string (format: `postgresql://postgres.[project-ref]:[password]@aws-0-[region].pooler.supabase.com:5432/postgres`). TARGET_DATABASE_URL is the shared project's direct Postgres connection string.
  </action>

**Confidence:** HIGH (primary sources: official Supabase docs, integration guide, actual codebase inspection)

## Summary

Phase 40 moves the Civic Trivia app from a standalone Supabase/Postgres database (schema `civic_trivia`) to a shared Supabase project that also hosts Empowered Accounts identity infrastructure. Three distinct tasks are required: (1) create the `trivia` schema with appropriate tables and RLS policies on the shared project, (2) migrate all 636 active question and collection records from the current database to the shared project, and (3) regenerate TypeScript types to include the new `trivia` schema and verify the build.

**Critical schema rename:** The existing codebase uses the schema name `civic_trivia`. The authoritative integration guide (`empowered-accounts-integration-guide.md`) and the phase requirements explicitly call for `trivia` as the schema name on the shared project. This is not just a content migration — the schema must be created under the new name `trivia`, and all Drizzle schema references in the codebase must be updated to match.

**Two new tables are needed that do not exist in the current schema:** `trivia.player_stats` (replaces the XP/stats columns currently on the local `users` table) and `trivia.player_prefs` (replaces the `timer_multiplier` column currently on the local `users` table). Both are keyed by UUID `user_id` referencing `public.users(id)`. The `question_flags.user_id` column is also currently `INTEGER` and must become `UUID` to reference `public.users(id)`.

**Primary recommendation:** Create the `trivia` schema on the shared Supabase project via a numbered migration file (`supabase/migrations/`), run the content migration as a one-time TypeScript script using `pg` connections between source and target, then regenerate types with `supabase gen types --linked --lang typescript --schema public,connect,empower,inform,trivia`.

**What:** All user-referencing columns use `UUID` referencing `public.users(id)` — never the old `INTEGER` users table

### Pattern 3: RLS Policies — Service Role Bypass + User-Scoped Reads

**What:** Enable RLS on all trivia tables. Service role key bypasses automatically. Authenticated users only see their own rows.
**When to use:** All `trivia` schema tables, per DB-04 requirement

```sql
-- Source: Supabase official RLS docs https://supabase.com/docs/guides/database/postgres/row-level-security

-- Enable RLS (must be done on ALL trivia tables)
ALTER TABLE trivia.player_stats    ENABLE ROW LEVEL SECURITY;
ALTER TABLE trivia.player_prefs    ENABLE ROW LEVEL SECURITY;
ALTER TABLE trivia.question_flags  ENABLE ROW LEVEL SECURITY;
-- Content tables (questions, collections, topics, etc.) — public reads, no user data
ALTER TABLE trivia.questions         ENABLE ROW LEVEL SECURITY;
ALTER TABLE trivia.collections       ENABLE ROW LEVEL SECURITY;
ALTER TABLE trivia.topics            ENABLE ROW LEVEL SECURITY;
ALTER TABLE trivia.collection_questions ENABLE ROW LEVEL SECURITY;
ALTER TABLE trivia.collection_topics    ENABLE ROW LEVEL SECURITY;
ALTER TABLE trivia.election_races       ENABLE ROW LEVEL SECURITY;

-- Content tables: public read, no write via client
CREATE POLICY "Public read for questions"
  ON trivia.questions FOR SELECT TO anon, authenticated USING (true);

CREATE POLICY "Public read for collections"
  ON trivia.collections FOR SELECT TO anon, authenticated USING (true);

CREATE POLICY "Public read for topics"
  ON trivia.topics FOR SELECT TO anon, authenticated USING (true);

-- User-scoped tables: owner only
CREATE POLICY "Users can read own stats"
  ON trivia.player_stats FOR SELECT TO authenticated
  USING ((SELECT auth.uid()) = user_id);

CREATE POLICY "Users can insert own stats"
  ON trivia.player_stats FOR INSERT TO authenticated
  WITH CHECK ((SELECT auth.uid()) = user_id);

CREATE POLICY "Users can update own stats"
  ON trivia.player_stats FOR UPDATE TO authenticated
  USING ((SELECT auth.uid()) = user_id)
  WITH CHECK ((SELECT auth.uid()) = user_id);

CREATE POLICY "Users can read own prefs"
  ON trivia.player_prefs FOR SELECT TO authenticated
  USING ((SELECT auth.uid()) = user_id);

CREATE POLICY "Users can upsert own prefs"
  ON trivia.player_prefs FOR INSERT TO authenticated
  WITH CHECK ((SELECT auth.uid()) = user_id);

CREATE POLICY "Users can update own prefs"
  ON trivia.player_prefs FOR UPDATE TO authenticated
  USING ((SELECT auth.uid()) = user_id)
  WITH CHECK ((SELECT auth.uid()) = user_id);

-- NOTE: service_role key automatically bypasses ALL RLS policies
-- No explicit service_role policies needed
```

**Critical:** Use `(SELECT auth.uid())` with a SELECT wrapper, not bare `auth.uid()`, for performance. This allows the query planner to cache the result per statement rather than re-evaluating per row.

### Pattern 4: Data Migration Script (TypeScript via `pg`)

**What:** Connect to source and target databases using `pg`, read content tables from source, insert into target
**When to use:** One-time content migration (questions, topics, collections, collection_questions, collection_topics, election_races)

```typescript
// Source: Supabase migration docs + standard pg Node.js pattern
// backend/src/scripts/migrate-to-shared-supabase.ts

import pg from 'pg';
const { Pool } = pg;

const source = new Pool({ connectionString: process.env.SOURCE_DATABASE_URL });
const target = new Pool({ connectionString: process.env.TARGET_DATABASE_URL });

async function migrateTable(tableName: string) {
  const { rows } = await source.query(
    `SELECT * FROM civic_trivia.${tableName} ORDER BY id`
  );
  if (rows.length === 0) return 0;

  // Build parameterized INSERT using COPY-style bulk insert
  const cols = Object.keys(rows[0]);
  const colList = cols.map(c => `"${c}"`).join(', ');

  // Use batch inserts with ON CONFLICT DO NOTHING for idempotency
  const batchSize = 500;
  let inserted = 0;
  for (let i = 0; i < rows.length; i += batchSize) {
    const batch = rows.slice(i, i + batchSize);
    const placeholders = batch.map((_, rowIdx) =>
      `(${cols.map((_, colIdx) => `$${rowIdx * cols.length + colIdx + 1}`).join(', ')})`
    ).join(', ');
    const values = batch.flatMap(row => cols.map(c => row[c]));
    await target.query(
      `INSERT INTO trivia.${tableName} (${colList}) VALUES ${placeholders}
       ON CONFLICT DO NOTHING`,
      values
    );
    inserted += batch.length;
  }
  return inserted;
}
```

  userId: uuid('user_id').primaryKey(),  // UUID, not serial

For `questionFlags.userId`, the type must change from `integer` to `uuid` (requires importing `uuid` from `drizzle-orm/pg-core`).

### Anti-Patterns to Avoid

- **Adding tables to `public`:** Never add trivia tables to the `public` schema. It is reserved for cross-feature infrastructure. Source: integration guide section 8.
- **Referencing `auth.users` directly:** Always reference `public.users(id)`, not `auth.users(id)`. Source: integration guide section 8.
- **Using service role client for reads:** Use per-request user-scoped client for reads that feed API responses; service role only for writes. Source: integration guide section 8.
- **Bare `auth.uid()` in RLS policies:** Always wrap in `(SELECT auth.uid())` for performance.
- **Skipping schema exposure:** Tables in custom schemas are invisible to the Supabase client unless the schema is added to Exposed Schemas in the Dashboard AND the grant SQL is run.
- **Drizzle push instead of Supabase CLI:** `drizzle-kit push` works for schema/table DDL but cannot manage RLS policies or schema exposure. Always use `supabase db push` for the shared project.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TypeScript types for trivia schema | Manual interface definitions | `supabase gen types` CLI | Types auto-sync with schema; manual types drift and cause runtime bugs |
| UUID generation | `uuid` npm package | `gen_random_uuid()` (Postgres built-in) | Supabase project has this built in; no extra dep needed |
| Multi-table atomic writes | Chained JS awaits | Postgres RPC (`SECURITY DEFINER` function) | Avoids partial writes; integration guide explicitly requires this pattern |
| Gem awards | Direct gem table inserts | `award_gems` RPC via service role | Platform gem ledger has history tracking; direct inserts bypass it |
| Role checking | Custom role table | `public.user_roles` via service role client | Platform role system is already built and audited |

**Key insight:** The Empowered platform has already built gems, roles, and user management. The trivia schema provides only trivia-specific isolation — it does not replicate any cross-feature infrastructure.

## Common Pitfalls

### Pitfall 1: Schema Name Mismatch (civic_trivia vs trivia)

**What goes wrong:** The current codebase uses `civic_trivia` as its schema name. The shared Supabase project must use `trivia`. If the migration script is written against `civic_trivia` and the Drizzle schema.ts is not updated, the backend will point at a schema that doesn't exist on the new database.
**Why it happens:** The schema was created as `civic_trivia` in Phase 13 before the shared platform architecture was decided. The integration guide post-dates that decision.
**How to avoid:** Sub-task 40-01 must (a) create `trivia` on the shared project, (b) update `schema.ts` to use `pgSchema('trivia')`, and (c) update `drizzle.config.ts` `schemaFilter` from `['civic_trivia']` to `['trivia']`.
**Warning signs:** TypeScript errors in `schema.ts`; `supabase gen types` output missing `trivia` namespace; backend returning 404 on game routes after cutover.

### Pitfall 2: Schema Not Exposed to PostgREST

**What goes wrong:** Tables are created in `trivia` schema but Supabase client queries return `PGRST106: The schema must be one of the following` error.
**Why it happens:** Custom schemas are not auto-exposed. PostgREST only exposes schemas listed in its `db_schema` config.
**How to avoid:** Two steps required: (1) run GRANT SQL in the migration file, (2) add `trivia` to Exposed Schemas in Supabase Dashboard → Settings → API. Both are required — neither alone is sufficient.
**Warning signs:** PGRST106 error on any `.schema('trivia')` query; schema visible in `psql` but not in Supabase Studio API docs.

### Pitfall 3: Service Role Contamination

**What goes wrong:** Service role client starts returning RLS errors or user-filtered data instead of bypassing RLS.
**Why it happens:** User JWT getting into the service role client's Authorization header (via SSR cookie sharing, `signUp()` call, or `createClient` options).
**How to avoid:** Keep separate client instances: one initialized with `service_role` key for admin operations, one initialized with user JWT for user-scoped reads. Never call auth methods on the service role client.
**Warning signs:** Service role client returns fewer rows than expected; RLS-related errors from queries that should bypass.

### Pitfall 4: Integer User IDs Leaking into New Schema

**What goes wrong:** `question_flags.user_id` is currently `INTEGER` (referencing the old local `users` table). If this column is migrated as-is, it will contain old integer IDs that don't map to any UUID in `public.users`.
**Why it happens:** The `question_flags` table was created in Phase 27 before the UUID migration was planned.
**How to avoid:** In sub-task 40-01, define `question_flags.user_id` as `UUID NOT NULL REFERENCES public.users(id)`. During Phase 40, no `question_flags` rows should be migrated (they reference old integer user IDs with no UUID equivalent). Start fresh in the new schema.
**Warning signs:** Foreign key constraint violations on insert; wrong type errors in Drizzle schema.

### Pitfall 5: Missing player_stats / player_prefs Tables

**What goes wrong:** Phase 41 (Auth integration) assumes `trivia.player_stats` and `trivia.player_prefs` exist. If Phase 40 only migrates content tables and forgets the new player tables, Phase 41 is blocked.
**Why it happens:** The tables don't exist in the current schema, so there's nothing to "migrate" — they must be created from scratch.
**How to avoid:** Sub-task 40-01 explicitly creates both tables as part of the DDL migration, even though they will start empty. Phase 41 populates them.
**Warning signs:** Phase 41 queries fail with "relation does not exist"; TypeScript compilation errors on `playerStats` or `playerPrefs` imports from schema.ts.

### Pitfall 6: Data Migration Idempotency

**What goes wrong:** Running the migration script twice creates duplicates (or unique constraint violations depending on how it fails).
**Why it happens:** Non-idempotent INSERT statements.
**How to avoid:** Use `INSERT ... ON CONFLICT DO NOTHING` for all content tables. Verify row counts before and after.
**Warning signs:** Primary key or unique constraint violations on second run; row counts doubling.

### Pitfall 7: SERIAL vs UUID Primary Keys

**What goes wrong:** Content tables (questions, collections, topics, etc.) currently use `SERIAL` (auto-increment integer) PKs. Migrating them to the new schema preserves integer IDs. The `player_stats` and `player_prefs` tables use UUID PKs. These are different things and must not be confused.
**Why it happens:** Content has no reason to switch to UUID. User-scoped tables MUST use UUID because they reference `public.users(id)` which is UUID.
**How to avoid:** Keep `SERIAL` for content tables (they have no user FK). Use `UUID PRIMARY KEY` only for `player_stats`, `player_prefs`, and `question_flags.user_id`.
**Warning signs:** Attempting to set `DEFAULT gen_random_uuid()` on content tables when they should stay `SERIAL`.

### Pitfall 8: Supabase CLI Not Linked

**What goes wrong:** `supabase db push` or `supabase gen types --linked` fails with "project not linked" error.
**Why it happens:** `supabase link` has not been run with the shared project's credentials.
**How to avoid:** Run `supabase link --project-ref <project-id>` before any CLI migration operations. Requires `SUPABASE_ACCESS_TOKEN` env var (personal access token from Supabase dashboard).
**Warning signs:** `Error: Cannot find project ref. Have you run supabase link?`

## Open Questions

1. **Supabase CLI `supabase link` credentials**
   - What we know: `supabase db push` and `supabase gen types --linked` require the CLI to be linked to the shared project via a personal access token
   - What's unclear: Whether the shared project's `SUPABASE_PROJECT_REF` and access token are already available in the project environment, or must be obtained from the project owner
   - Recommendation: The planner should include a prerequisite step to obtain/confirm shared project credentials before any migration tasks begin

2. **Shared project `public.users` table structure**
   - What we know: The integration guide confirms `public.users(id)` is UUID; all FKs must reference it
   - What's unclear: The exact column structure of `public.users` (beyond `id UUID`). Phase 40 only needs `id` for FK reference, but later phases (41+) will need to know if `display_name` or other columns exist
   - Recommendation: This is fine for Phase 40 — we only need `public.users(id)` to exist. Flag as open for Phase 41 research.

3. **`question_flags` data migration decision**
   - What we know: Existing `question_flags` rows have integer `user_id` values (old local users table). These integer IDs have no UUID equivalent in `public.users`.
   - What's unclear: Whether admin-side flag data from the old system should be preserved at all, or discarded as it cannot be attributed to UUID users.
   - Recommendation: Do NOT migrate `question_flags` rows. Start fresh. The table structure changes (integer → UUID), so old rows are invalid. Document this in the migration script.

4. **`supabase/` directory initialization**
   - What we know: `supabase db push` requires a `supabase/config.toml` and `supabase/migrations/` directory at repo root. Neither currently exists.
   - What's unclear: Whether `supabase init` needs to run first, and if the `[api] schemas` list in `config.toml` must include `trivia` for local development parity.
   - Recommendation: Run `supabase init` as the first step in 40-01, then `supabase link`, before writing migrations. Include `trivia` in the `config.toml` api schemas list.

5. **Source database connection during migration**
   - What we know: The current Supabase dev project database URL is in `.env` as `DATABASE_URL`. The shared project has a separate connection string.
   - What's unclear: Whether the current dev project's direct Postgres connection URL is available (vs the Supabase pooler URL). pg_dump works best with direct connections.
   - Recommendation: Use two env vars: `SOURCE_DATABASE_URL` (current dev project) and `TARGET_DATABASE_URL` (shared project). Both Supabase projects expose a direct connection string at `db.<project-ref>.supabase.co:5432`.

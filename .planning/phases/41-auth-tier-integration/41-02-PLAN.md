**sessionService.ts plausibility fix** (line ~256):

The current code does:
```typescript
const user = await User.findById(session.userId as number);
const timerMultiplier = user?.timerMultiplier ?? 1.0;
```

`User.findById` takes an integer. After Phase 41, `session.userId` is a UUID string for authenticated users. Casting a UUID string to `number` produces `NaN`, which will cause the query to fail.

**Fix:** Before the `User.findById` call, check if `session.userId` is a UUID string. If so, skip the lookup and default to `timerMultiplier = 1.0`:

```typescript
// Phase 41: UUID users skip User.findById (takes integer ID)
// timerMultiplier defaults to 1.0 — player_prefs migration is Phase 43
let timerMultiplier = 1.0;
if (typeof session.userId === 'number') {
  // Legacy integer user path (will be removed in Phase 44)
  const user = await User.findById(session.userId);
  timerMultiplier = user?.timerMultiplier ?? 1.0;
}
// For UUID string users: timerMultiplier stays 1.0
```

This preserves backward compatibility for any hypothetical integer-userId sessions still in Redis (there won't be any after cutover, but defensive coding is free).

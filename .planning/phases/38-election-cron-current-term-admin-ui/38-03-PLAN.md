  <action>
Completely rewrite `frontend/src/pages/admin/ElectionsPage.tsx`. This is a full rewrite — the existing file has a flat table and basic generation modals. The new version needs a three-tab lifecycle UI.

**CRITICAL: headlessui v2 API.** The installed version is `@headlessui/react` ^2.2.9 (v2). Use named exports:
```tsx
import { Tab, TabGroup, TabList, TabPanel, TabPanels, Dialog, DialogPanel, DialogTitle, Transition, TransitionChild } from '@headlessui/react';
```
Do NOT use `Tab.Group`, `Tab.List`, `Tab.Panel`, `Dialog.Panel`, `Dialog.Title`, `Transition.Child` — those are v1 compound patterns that don't work in v2.

**Data fetching:**
- Replace the existing `fetchRaces` (which calls `GET /election-races`) with a call to `GET /api/admin/election-races/classified`.
- Response shape: `{ activeElections: Race[], pendingGeneration: Race[], awaitingFollowup: Race[], cronLastRun: { timestamp: string; racesDetected: number; processed: number; failures: number } | null }`.
- Store in state: `activeElections`, `pendingGeneration`, `awaitingFollowup`, `cronLastRun`.
- Each race in the response includes `questionCount` (number of active+draft questions).

**Overall page structure:**
1. Page header: "Election Management" title + "New Election Race" button (preserves existing create form toggle).
2. Cron last-run banner: if `cronLastRun` is not null, show a small info bar: "Last cron run: {relative time} — {processed} races processed{, {failures} failures}" with appropriate coloring (green if no failures, amber if failures). If null, show "Cron: No runs recorded since last server restart" in gray.
3. Create form: preserve the existing create race form exactly as-is (it's well-built). Keep all the same fields, validation, and candidate management. After successful creation, re-fetch classified data.
4. Tab bar using headlessui v2 `TabGroup`:
   - Tab 1: "Active Elections ({count})"
   - Tab 2: "Pending Generation ({count})"
   - Tab 3: "Awaiting Follow-up ({count})"
   - Tab styling: use admin red theme. Selected tab: `border-b-2 border-red-700 text-red-700 font-semibold`. Unselected: `text-gray-500 hover:text-gray-700`.

**Tab 1: Active Elections**
Each race row shows:
- Seat name (bold) + election date + jurisdiction + question count badge + status badge ("Active" in green)
- Candidates summary (count + names in gray)
- "Re-generate" button (amber/warning styled)

Re-generate flow:
1. Click "Re-generate" -> fetch `GET /election-races/{id}/question-count` to get activeCount and draftCount.
2. Show confirmation modal (Dialog): "Re-generating will archive {activeCount} active questions and delete {draftCount} draft questions for this race. Continue?"
3. On confirm -> `POST /election-races/{id}/regenerate`.
4. Show loading state during generation (reuse loading modal pattern).
5. On success: show success toast with new question count, re-fetch classified data.
6. On error: show error in a modal.

**Tab 2: Pending Generation**
Each race row shows:
- Seat name (bold) + election date + jurisdiction + status badge ("Pending" in amber)
- Candidates summary
- Three action buttons:
  - "Generate Questions" (red primary) — opens the existing collection slug prompt modal, then starts generation via `POST /election-races/{id}/generate`.
  - "Edit Race" — opens edit modal (pre-populated form, same fields as create) -> `PUT /election-races/{id}`.
  - "Delete Race" — confirmation modal: "Delete {seat}? Questions linked to this race will be unlinked (not deleted)." -> `DELETE /election-races/{id}`.

The existing generation flow (collection prompt modal -> loading modal -> success/blocked/error modals) should be preserved from the current implementation. It works well — just move it to only appear for pending generation tab actions and add the re-generate flow for active tab.

**Tab 3: Awaiting Follow-up**
Each race row shows:
- Seat name (bold) + election date + jurisdiction + status badge ("Expired" in gray)
- Candidates summary
- "Enter Result" button (red primary)

Enter Result flow:
1. Click "Enter Result" -> open a Dialog modal with:
   - Modal title: "Enter Election Result — {seat}"
   - Winner name text input (required)
   - Term end date input (type="date", required)
   - Collection slug dropdown (same pattern as generation prompt — fetch from /api/game/collections, auto-match from jurisdiction)
   - "Generate Current-Term Questions" submit button
   - "Cancel" button
2. On submit -> `POST /election-races/{id}/enter-result` with `{ winnerName, termEndDate, collectionSlug }`.
3. Show loading modal during generation.
4. On success: show success toast: "{N} current-term questions created for {jurisdiction}", re-fetch classified data (race should move out of Awaiting Follow-up).
5. On 409 (already generated): show info modal "Current-term questions already generated for this race."
6. On error: show error modal.

**Status badges:**
- "Active": `bg-green-100 text-green-700` (questions live, election upcoming)
- "Pending": `bg-amber-100 text-amber-700` (awaiting generation)
- "Expired": `bg-gray-100 text-gray-600` (election passed, needs follow-up)

**Toast notifications:**
Use simple in-page toast pattern (no library):
```tsx
const [toast, setToast] = useState<{ message: string; type: 'success' | 'error' } | null>(null);
const showToast = (message: string, type: 'success' | 'error' = 'success') => {
  setToast({ message, type });
  setTimeout(() => setToast(null), 4000);
};
```
Render as fixed-position element bottom-right. Green for success, red for error.

**Race row component:**
Extract a `RaceRow` helper component (can be defined inside `ElectionsPage` or as a local function) to avoid repeating the seat/date/jurisdiction/candidates layout across all three tabs. Pass `actions` as a prop or render prop.

**Edit Race modal:**
Same fields as the create form (seat, electionType, electionDate, timezone, jurisdiction dropdown, candidates). Pre-populate from the selected race. On save: `PUT /election-races/{id}` with only changed fields. On success: re-fetch classified, close modal, show success toast.

**Empty states:**
Each tab should show a friendly empty state when it has no races:
- Active: "No active elections with generated questions."
- Pending: "No elections pending generation."
- Awaiting: "No expired elections awaiting follow-up."

**Important implementation notes:**
- All fetch calls must include `Authorization: Bearer ${accessToken}` header.
- All POST/PUT/DELETE calls must include `Content-Type: application/json` header.
- The `accessToken` comes from `useAuthStore()` (existing pattern).
- `API_URL` comes from `'../../services/api'` (existing import).
- The Dialog `onClose` for the loading modal should be a no-op `() => {}` to prevent closing during generation.
- After any mutation (create, generate, re-generate, enter result, edit, delete), re-fetch the classified endpoint to update all tabs.

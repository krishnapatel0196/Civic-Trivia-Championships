  <action>
Create `backend/src/services/DuplicateReviewService.ts`:

1. **loadReport(reportPath?: string)** - Load the most recent scanner JSON report from `.planning/dedup-reports/`. If no path specified, find the latest `duplicates-*.json` file by timestamp in filename. Parse the JSON and detect the format using backward-compatible logic:

   ```typescript
   const parsed = JSON.parse(raw);
   let clusters: DuplicateCluster[];
   let advancedFlags: AdvancedFlag[];
   if (Array.isArray(parsed)) {
     // Old format: bare array of clusters (pre-Plan 32-04)
     clusters = parsed;
     advancedFlags = [];
   } else if (parsed && typeof parsed === 'object' && 'clusters' in parsed) {
     // New format: { clusters: [...], advancedFlags: [...] } (from Plan 32-04)
     clusters = parsed.clusters;
     advancedFlags = parsed.advancedFlags ?? [];
   } else {
     throw new Error('Unrecognized scanner report format');
   }
   ```

   Store both `clusters` and `advancedFlags` as instance properties. This is critical because Plan 32-04 (running in the same wave) changes the JSON report format from a bare array to `{ clusters: [...], advancedFlags: [...] }`. The service must handle both formats from the start so execution order between Plan 01 and Plan 04 does not matter.

   Also load question DB IDs by mapping externalId to id via a single DB query.

2. **getClusters(filter?: { tier?: SimilarityTier, resolved?: boolean })** - Return clusters from the loaded report, optionally filtered by tier or resolution status. Track which clusters have been resolved in an in-memory Map<string, ClusterResolution> where ClusterResolution = { keepId: string, archivedIds: string[], resolvedAt: Date, archivedDbIds: number[] }. Include auto-resolve recommendations for clusters where the highest similarity is >= 0.90 (mark as `autoResolvable: true` in the response).

3. **getAdvancedFlags()** - Return the `advancedFlags` array loaded from the report. Also expose a method **getFlagsForQuestion(externalId: string)** that filters advancedFlags to return only flags where `questionA === externalId || questionB === externalId`.

4. **resolveCluster(clusterId: string, keepExternalId: string)** - Validate that keepExternalId exists in the cluster. Use a Drizzle transaction to:
   - Set status='archived' on all cluster questions EXCEPT the kept one
   - Append history entry { action: 'archived', timestamp: ISO, reason: 'duplicate-review' } to expirationHistory
   - Track the resolution in the in-memory map
   - Return the list of archived externalIds for JSON sync

5. **undoResolve(clusterId: string)** - Reverse a resolution: Drizzle transaction to restore archived questions to status='active', remove from resolution tracking. Only allow undo if resolved within the last 30 seconds (matching Gmail/GitHub undo patterns).

6. **autoResolveAll()** - Batch-resolve all clusters with highest similarity >= 0.90, using the ClusterBuilder's existing recommendation (which already applies Federal > State > City hierarchy and quality score tiebreaker). Return summary of how many clusters resolved and which questions archived.

Create `backend/src/services/JSONSyncService.ts`:

1. **syncAfterArchive(archivedExternalIds: string[])** - For each JSON source file in `backend/src/data/`:
   - Read the file
   - Parse the questions array
   - Filter out any questions whose `id` field (the externalId in JSON files) matches the archived list
   - Write back the filtered array with proper formatting (2-space indent, trailing newline)
   - Return a summary of changes per file (e.g., { file: 'questions.json', removed: 3, remaining: 114 })

2. **syncAfterRestore(restoredExternalIds: string[])** - This is harder since we need the full question data. For undo, query the DB for the full question records and re-insert into the appropriate JSON files based on collection membership. Use the collection name to determine which JSON file (Federal Civics -> questions.json, Bloomington IN -> bloomington-in-questions.json, etc.).

Map collection names to JSON files:
- 'Federal Civics' -> 'questions.json'
- 'Bloomington, IN' -> 'bloomington-in-questions.json'
- 'Los Angeles, CA' -> 'los-angeles-ca-questions.json'
- 'Indiana' -> 'indiana-state-questions.json'
- 'California' -> 'california-state-questions.json'
- 'Fremont, CA' -> 'fremont-ca-questions.json'

IMPORTANT: Only sync JSON files AFTER successful DB transaction. Never write JSON before DB commit succeeds.

  <action>
Add the following endpoints to `backend/src/routes/admin.ts` (after existing flag endpoints, before the export):

1. **GET /duplicates** - Load and return duplicate clusters with advanced flags
   - Query params: `tier` (optional: 'exact' | 'near-duplicate' | 'possible'), `resolved` (optional: 'true' | 'false')
   - Creates a DuplicateReviewService instance, calls loadReport() then getClusters(filter)
   - Also calls getAdvancedFlags() to include in the response
   - Returns: `{ clusters: EnrichedCluster[], advancedFlags: AdvancedFlag[], summary: { total, resolved, pending, autoResolvable } }`
   - Each EnrichedCluster adds `autoResolvable: boolean`, `resolved: boolean`, `resolution?: ClusterResolution` to the base DuplicateCluster type
   - The `advancedFlags` array contains all flags from the report (answer-leakage, same-source-cluster, inverse-duplicate). The frontend uses this to display flag badges on affected questions.

2. **POST /duplicates/resolve** - Resolve a single cluster
   - Body: `{ clusterId: string, keepExternalId: string }`
   - Validates with Zod schema
   - Calls DuplicateReviewService.resolveCluster()
   - Then calls JSONSyncService.syncAfterArchive() with the archived externalIds
   - Returns: `{ success: true, archived: string[], jsonSync: SyncSummary[] }`

3. **POST /duplicates/auto-resolve** - Auto-resolve all 90%+ clusters
   - No body required
   - Calls DuplicateReviewService.autoResolveAll()
   - Then calls JSONSyncService.syncAfterArchive() with all archived externalIds
   - Returns: `{ success: true, clustersResolved: number, totalArchived: number, jsonSync: SyncSummary[] }`

4. **POST /duplicates/undo** - Undo a cluster resolution
   - Body: `{ clusterId: string }`
   - Validates with Zod schema
   - Calls DuplicateReviewService.undoResolve()
   - Then calls JSONSyncService.syncAfterRestore() with the restored externalIds
   - Returns: `{ success: true, restored: string[] }` or `{ error: 'Undo window expired' }` with 400 status

5. **GET /duplicates/summary** - Quick stats without loading full clusters
   - Returns collection-level duplicate counts from the loaded report
   - Shape: `{ collections: { name: string, questionCount: number, duplicateCount: number }[], totalClusters: number, totalDuplicates: number }`

All endpoints follow existing admin.ts patterns:
- Wrapped in try/catch with console.error + 500 response
- Protected by the existing `router.use(authenticateToken, requireAdmin)` middleware
- Use Zod for body validation (already imported)

NOTE: The DuplicateReviewService should be instantiated as a singleton per request lifecycle since it holds in-memory state. Use a module-level instance that persists across requests (lazy-initialized on first GET /duplicates call). This is acceptable for a single-admin tool — no concurrent user concerns.

### OPEN GAP: Admin Access After Fresh Login

**Phases:** 43 (Login.tsx) ↔ 45 (AdminGuard with tierResolved)

**Description:**
Phase 45 added `tierResolved` state to `authStore.ts` to prevent `AdminGuard` from rendering until the user's tier has been authoritatively resolved from the accounts API. `AuthInitializer.tsx` correctly calls `setTierResolved(true)` in all 4 of its exit paths (lines 21, 49, 55, 62), covering the page-reload flow.

However, `Login.tsx` and `Signup.tsx` call `setAuth(token, user)` after successful authentication but do **not** call `setTierResolved(true)`. The `setAuth` action in `authStore.ts` also does not set `tierResolved: true`. After a fresh login (first page load, no prior session), `AuthInitializer` only handles the refresh-token path — if no refresh token exists yet, it exits early (line 21: `setTierResolved(true)` before returning). But if the user just logged in and `AuthInitializer` already ran on app mount before the login, it does not re-run after the login completes.

**Failure mode:**
1. User opens app → `AuthInitializer` runs → no refresh token in localStorage yet → `setTierResolved(true)` called at early-return path (line 21)
2. User navigates to `/login` → submits credentials → `authService.login()` → `setAuth(token, user)` called → `tierResolved` is now `false` (clearAuth resets it; setAuth doesn't set it back)
3. User (admin) navigates to `/admin` → `AdminGuard` evaluates `isLoading || !tierResolved` → `tierResolved` is `false` → returns `null` → blank screen indefinitely

Wait — re-examining: `clearAuth` resets `tierResolved: false`. But `setAuth` is called on login (not `clearAuth`). Does `setAuth` reset `tierResolved`? Let me trace:

- Fresh app load: `tierResolved` starts as `false` (initial store state)
- `AuthInitializer` runs: no refresh token → early return → `setTierResolved(true)` → `tierResolved: true`
- User goes to `/login`, logs in → `Login.tsx` calls `setAuth(accessToken, user)` → `setAuth` in `authStore.ts` does NOT reset `tierResolved`
- So after login, `tierResolved` is still `true` from `AuthInitializer`'s early-return path

**Revised scope:** The gap occurs if a user starts with an EXPIRED session (clearAuth was called, resetting `tierResolved: false`) and then logs in. After `clearAuth`, `tierResolved: false`. After `Login.tsx` → `setAuth(...)`, `tierResolved` remains `false`. AdminGuard returns null indefinitely until the next page reload (which triggers AuthInitializer again).

**Severity:** Medium. Affects admin users whose session expired mid-use. Workaround: page reload after logging back in. Not a first-login regression.

**Fix:** `setAuth` in `authStore.ts` should set `tierResolved: true`, or `Login.tsx`/`Signup.tsx` should call `setTierResolved(true)` after `setAuth`. Since `setAuth` is the natural place to establish a complete auth state, updating `setAuth` to include `tierResolved: true` is the cleanest fix.
